# MagicCvetov - Вопросы по архитектуре

## 🚨 КРИТИЧЕСКИЕ ВОПРОСЫ ПО TELEGRAM MINI APP (27.01.2025)

### В1: Почему пользователи не могут открыть Telegram Mini App в продакшне?

**Ответ:** Найдена критическая проблема с путями к статическим файлам:
- ❌ Telegram бот ссылается на `/static/miniapp/checkout.html` → HTTP 403 Forbidden
- ✅ Mini App доступен по альтернативному пути `/miniapp/checkout.html` → HTTP 200 OK
- 🔧 **РЕШЕНИЕ**: Обновить ссылку в @DIMBOpizzaBot на рабочий путь

### В2: Почему записи не попадают в таблицу telegram_auth_tokens?

**Ответ:** Цепочка проблем из-за недоступности Mini App интерфейса:
1. Пользователи не могут загрузить Mini App через бота (HTTP 403)
2. JavaScript код авторизации не выполняется
3. `requestContact()` не вызывается автоматически
4. API `enhanced-auth` никогда не получает запросы
5. Записи в `telegram_auth_tokens` не создаются

### В3: Как система кросс-платформенной авторизации должна работать?

**Ответ:** Архитектура уже полностью реализована:
- Пользователь открывает Mini App → initData от Telegram
- requestContact автоматически → Получение номера телефона
- POST /api/v1/telegram-webapp/enhanced-auth → Создание User в БД
- Сохранение в telegram_auth_tokens → JWT токен для всех платформ

### В4: Какие компоненты уже готовы и работают?

**Ответ:** ✅ Готово 95% системы:
- ✅ Backend API: TelegramWebAppService.enhancedAuthenticateUser()
- ✅ Enhanced-auth эндпоинт: `POST /api/v1/telegram-webapp/enhanced-auth`
- ✅ JavaScript: автоматический requestContact() с fallback
- ✅ База данных: таблица telegram_auth_tokens
- ✅ Версионная совместимость: Telegram API 6.0 и 7.7
- ❌ **БЛОКЕР**: Доступ к Mini App в продакшне

### В5: Что нужно исправить для запуска системы?

**Ответ:** Одна критическая задача:
1. **НЕМЕДЛЕННО**: Обновить URL в @DIMBOpizzaBot
   - Текущий: `https://api.dimbopizza.ru/static/miniapp/checkout.html`
   - Рабочий: `https://api.dimbopizza.ru/miniapp/checkout.html`
2. Протестировать авторизацию через Mini App
3. Проверить создание записей в `telegram_auth_tokens`

**Ожидаемое время устранения**: 5-10 минут

## Вопросы по масштабированию

1. **Горизонтальное масштабирование:**
   - Какие компоненты системы должны масштабироваться в первую очередь?
    В первую очередь должны масштабироваться: API-серверы, обрабатывающие пользовательские запросы; сервисы обработки заказов; база данных (через репликацию для чтения). Компоненты, работающие с файлами и изображениями, также должны быть готовы к масштабированию.

   - Требуется ли поддержка нескольких регионов (географическое распределение)?
   нет
   - Какая стратегия шардирования данных предпочтительна?
   нет

1. **Микросервисы:**
   - По каким критериям следует разделять монолит на микросервисы в будущем?
   Критерии разделения: бизнес-домены (управление пользователями, каталог, корзина, заказы, оплата, доставка); различная интенсивность нагрузки на компоненты; разные требования к масштабированию; отдельные команды разработки для разных компонентов.

   - Какая стратегия межсервисного взаимодействия предпочтительна (REST, gRPC, сообщения)?

   Гибридный подход: синхронное взаимодействие через REST/gRPC для запросов, требующих немедленного ответа; асинхронное взаимодействие через сообщения (Kafka/RabbitMQ) для событий и операций, не требующих мгновенного ответа.

   - Как организовать согласованность данных между сервисами?
   Использование паттерна Saga для транзакций, затрагивающих несколько сервисов; событийно-ориентированная архитектура для синхронизации состояний; возможное применение паттерна CQRS с отдельными моделями для чтения и записи.
